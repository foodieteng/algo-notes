重點回顧// Concept Recap

關鍵點內容
重新框架 permutation ⇒ 兩兩相異 ⇒ good ⟺ max − min = R − L，把連續性壓成 O(1) 等式。
分治結構 切半 → 遞迴左右半 → 跨界依 max/min 位置分 4 case 各自 O(N) 計數。
4 case 分類 (max, min) ∈ {LL, RR, LR, RL}：前兩種「位置已決定 ⇒ 反推 R 或 L」；後兩種「等式 hashing + two-pointer」。
嚴格不等式 排列保證 lMax[L] ≠ rMax[R]、lMin[L] ≠ rMin[R]，所以 case 之間嚴格互斥不重複。
複雜度 T(N) = 2·T(N/2) + O(N) = Θ(N log N)；跨 T 組為 O(S log S)。
型別 答案上限 ≈ N(N+1)/2 = 1.25 × 10¹¹，必用 long long。

觀念題// Concept Questions

Q1 重新框架

為何 distinct integers 的集合 S 可以用 max − min == |S| − 1 判斷是否「排序後連續」？

Q2 分治結構

為什麼跨界部分要再分 4 case？單看 max − min == R − L 不能一次算完嗎？

Q3 等式推導

Case 3（max 在左、min 在右）為何用 L + lMax[L] == R + rMin[R] 當 cnt 索引？從 goodness 條件推導一遍。

Q4 monotone

為什麼 lMax[L] 從 mid 往左掃時是 monotone non-decreasing？對 lMin[L] 也成立嗎？

Q5 推廣

如果輸入不是 permutation（值可能重複），這個 D&C 還能正確運作嗎？需要怎樣修改？

App 3 改題

// ANSWERS

先自己想過再對答案 — Q 對應 Q、App 對應 App。

A1 · 重新框架

設 S 是 |S| 個 distinct 整數的集合，記 m = min(S)、M = max(S)。

下界：要塞 |S| 個不同整數進 [m, M] 這個區間，至少需要長度 |S| − 1，即 M − m ≥ |S| − 1。這是 pigeonhole 風格的下界 — 不同的東西之間至少差 1。

等號成立 ⟺ 連續：M − m = |S| − 1 意味著區間 [m, M] 剛好容納 |S| 個整數， 而 S 已經有 |S| 個不同整數塞進去 ⇒ S = {m, m+1, …, M}，排序後就是連續整數。

把這個套到 S = {a[L], …, a[R]}：|S| = R − L + 1，所以 good ⟺ max − min = R − L。

A2 · 分治結構

max − min == R − L 是 O(1) 判定，但前提是我們已經知道 max 與 min。 當 (L, R) 跨界時，max 可能來自左半或右半、min 也可能來自左半或右半 — 4 種組合。

分 4 case 的好處：每個 case 內部 max 與 min 的來源固定了，可以寫成關於 L, R 的等式：

  • Case 1/2：max 與 min 都來自同一半 ⇒ 對 L（或 R）來說 max 與 min 已知 ⇒ R（或 L）反推出來，O(1) 驗證。
  • Case 3/4：max 與 min 分別來自兩半 ⇒ goodness 變成「左半某函數 == 右半某函數」⇒ 用 bucket[] 做等式 hashing。

不分 case 的話 max/min 的位置不明，無法把條件化為等式，回到 O(N²)。

A3 · 等式推導

Case 3 的設定：max 在左半 ⇒ 跨界區間的 max = lMax[L]； min 在右半 ⇒ 跨界區間的 min = rMin[R]。

goodness 條件：

// CASE 3 derivation
max(a[L..R]) − min(a[L..R]) = R − L
        lMax[L] − rMin[R] = R − L
        lMax[L] + L       = R + rMin[R]
       └─── 左半函數 ───┘   └── 右半函數 ──┘

所以對每個 L 算出 key = L + lMax[L]，去 bucket[] 查有多少 R 也產出相同 key（且滿足前提條件 lMax>rMax、rMin<lMin）。

bucket[] 用「R + rMin[R]」當 index 累積；查詢用「L + lMax[L]」。兩邊的 index 是同一個量（key），等號自動成立。

A4 · monotone

lMax[L] 定義為 max(a[L], a[L+1], …, a[mid])。當 L 往左走（L 減小），多納入一個元素 a[L−1]， lMax[L−1] = max(lMax[L], a[L−1])，所以 lMax[L−1] ≥ lMax[L] — 是 monotone non-decreasing。

同理 lMin[L] = min(lMin[L+1], a[L])，lMin[L−1] ≤ lMin[L] — monotone non-increasing。

這是 two-pointer 能用的關鍵：L 由 mid 往 lo 走時，lMax 與 lMin 都單調，所以「哪些 R 滿足 max-side / min-side」 是個單調擴張或收縮的集合，maxPtr / minPtr 只往一個方向走 ⇒ 全程 O(N)。

A5 · 推廣

有重複值時，max − min == R − L 不再等價於 good。例如 [1, 1, 2]： max=2、min=1、R−L=2，但元素集合是 {1, 1, 2}，不是連續整數（題目改成 sorted & consecutive distinct values 也不行，因為有重複）。

更精確：lemma 第 1 步「|S| 個 distinct 整數 ⇒ max − min ≥ |S| − 1」用到了 distinct 前提。若有重複， |distinct(S)| < R − L + 1，max − min 可能比 R − L 小。

修改方案：condition 改成「max − min == #distinct − 1 AND #distinct == R − L + 1」。 第二個條件等於說「沒有重複值」。這需要額外維護 #distinct，整題就不是這個 D&C 套路了 — 更可能用莫隊算法或滑動視窗。

A · App 1

已排序 [1, 2, 3, 4, 5]：任意連續子陣列 a[L..R] 都是 {L, L+1, …, R}，自然是連續整數 ⇒ 全部都是 good。

good (L, R) 數量 = 所有 1 ≤ L ≤ R ≤ N = C(N+1, 2) = N(N+1)/2。

N = 5 ⇒ 5·6/2 = 15 ✓ 符合輸出。

N = 5 × 10⁵ ⇒ 5 × 10⁵ × (5 × 10⁵ + 1) / 2 ≈ 1.25 × 10¹¹ ⇒ 超出 int 上限，必須 long long。

A · App 2

T = 5 × 10⁵, N = 1：每組 solve(1, 1) 直接回傳 1，沒有遞迴。每組 O(1) ⇒ 總共 O(T) = O(5 × 10⁵)。

單組 N = 5 × 10⁵：O(N log N) ≈ 10⁷。

兩種情境的總工時都受限於 S ≤ 5 × 10⁵：

  • T 多 N 小：總工時 = T × O(1) = O(T) = O(S)
  • T 少 N 大：總工時 = O(N log N) = O(S log S)

結論：「N 大那邊比較吃緊」 — O(S log S) 比 O(S) 多了 log 因子。實務上 T 雖然多，但每組 IO 跟 main 迴圈開銷 可能比演算法本身還貴 — 所以兩邊都不能忽視 fast IO。

A · App 3

改為「輸出所有 good (L, R)」：

  • 每個 case 找到 good 時，不只 ++ans，還要把 (L, R) 記錄下來（push 到 vector）。
  • Case 1/2：每個 L (或 R) 找到時直接 emit；O(1) per match。
  • Case 3/4：bucket[] 只能告訴你「有幾個 R 配對」，不能直接 emit 配對細節。需要把 bucket[] 換成 vector<int> bucket[2*N+5]，存所有 R；查詢時 iterate bucket，emit 每個 (L, R)。

複雜度：

  • 找完所有 good 的「工時」變成 O(N log N + K)，K 是 good 區間總數，最壞 K = N(N+1)/2 = O(N²)。
  • 所以最壞情形 輸出本身就 O(N²)，但這不是演算法的成本 — 是輸出量的下界（你要 print N² 行）。

D&C 的優勢只在計數能輸出 O(N²) 結果只需 O(N log N) 時間；要列舉就無可避免地要付出 O(K) 輸出成本。
