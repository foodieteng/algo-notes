重點回顧// Concept Recap

關鍵點內容
問題本質 「攻 b 前先攻 a」= 有向邊 a→b ⇒ 求拓樸排序。
字典序最小 Kahn 把 queue 換成 min-heap，每步取可選中最小編號。
解鎖機制 取出 u 後對每條 u→w 做 --indeg[w]，降到 0 才進堆。
循環判定 輸出個數 ≠ n ⇒ 有環 ⇒ 輸出 QAQ，不必額外偵測。
貪心正確性 交換論證：第一格非「最小可選」不可，逐格歸納 ⇒ 全域字典序最小。
複雜度 O((n + m) log n)，每點進出堆一次、每邊掃一次。

觀念題// Concept Questions

Q1 資料結構

普通 Kahn 用一般 queue 也能拓樸排序，為什麼本題一定要換成 min-heap？

Q2 邊方向

輸入「a b」代表攻 b 前先攻 a。建圖時邊該朝哪個方向？indeg 加在誰身上？

Q3 循環偵測

程式沒有寫任何「找環」的程式碼，卻能正確輸出 QAQ。它是怎麼判斷無解的？

Q4 貪心反例

「把答案最後排序一下不就字典序最小了？」這個想法為什麼錯？舉個會壞掉的例子。

Q5 正確性

為什麼「每步取目前可選的最小」能保證整個序列字典序最小，而不只是區域最優？

範例 Trace · 跑一次 code 看會輸出什麼// Worked Trace

取一個能看出「小編號被迫延後」的例子：n = 5，兩條邊 0→1、3→2。

// INPUT1
5 2
0 1
3 2

建圖後：

陣地 v01234
adj[v]{1}{}{}{2}{}
indeg[v]01100
 
// SEED · indeg=0 進堆
heap = {0, 3, 4}    ← 1 欠 0、2 欠 3，先不能進
order = []
// STEP 1 · pop 0
取最小 → u = 0
order = [0]
鬆弛 adj[0] = {1}: --indeg[1] = 0 → push 1
heap = {1, 3, 4}
// STEP 2 · pop 1
取最小 → u = 1
order = [0, 1]
adj[1] = {} → 無解鎖
heap = {3, 4}
// STEP 3 · pop 3
取最小 → u = 3    ← 注意：2 還沒進堆（還欠 3），所以取 3 不是 2
order = [0, 1, 3]
鬆弛 adj[3] = {2}: --indeg[2] = 0 → push 2
heap = {2, 4}
// STEP 4 · pop 2
取最小 → u = 2    ← 2 終於解鎖，才被排進來（在 3 之後）
order = [0, 1, 3, 2]
heap = {4}
// STEP 5 · pop 4
取最小 → u = 4
order = [0, 1, 3, 2, 4]
heap = {}  → 結束，order.size() = 5 = n ✓
// OUTPUT0 1 3 2 4
觀察：2 編號比 3 小，卻排在 3 後面 — 因為在「該不該選 2」的那一刻，2 還欠著前置 3，根本不在可選集合裡。 貪心取的是「目前可選的最小」，不是「全場最小」。

應用題// Quick Calculations

App 1 手算

官方樣本第一組 n=3、邊 0→1, 1→2, 0→2，請手動跑出輸出。第二組 0→1, 1→2, 2→0 又是什麼？

App 2 資料量

m 可達 3×10⁵、再 ×5 組測資。若忘了關 sync_with_stdio 或用了 endl，會發生什麼？

App 3 變形

若題目改成要「字典序最大」的拓樸序，程式要怎麼改？只動一行可以嗎？

// ANSWERS

先自己想過再對答案 — Q 對應 Q、App 對應 App。

A1 · 資料結構

一般 queue 的 Kahn 只能保證輸出「某一個合法拓樸序」— 出隊順序取決於入隊先後， 無法控制字典序。本題要的是字典序最小的那一個，必須在每一步都從「目前可選的點」裡 挑編號最小者。能在動態變化的集合裡反覆取最小值的資料結構，正是 min-heap（小根堆） ⇒ priority_queue<int, vector<int>, greater<int>>。

A2 · 邊方向

「攻 b 前先攻 a」表示 a 是 b 的前置條件，邊朝 a → b。 b 多了一個必須先完成的對象 ⇒ indeg[b]++。 

// edge (a, b)
adj[a].push_back(b);   // a 完成 → b 解鎖一步
indeg[b]++;            // b 的「欠債數」+1

若把方向寫反成 b → a，整個依賴關係顛倒，輸出會是錯的（通常變成「答案的反序」）。

A3 · 循環偵測

靠「輸出個數」隱式判斷。一個點只有在 indeg 降到 0 時才會進堆； 若它在一個環裡，環上每個點都互相欠著對方，沒有人能先降到 0，於是它們永遠進不了堆。

堆空、迴圈結束時，這些環上的點都沒被輸出 ⇒ order.size() < n ⇒ 判定無解、印 QAQ。 所以不需要額外的 DFS 找環。

A4 · 貪心反例

「最後把答案排序」會破壞合法性，根本不是同一個問題。看本頁 trace 的例子，正確答案是 0 1 3 2 4；若硬把它排序成 0 1 2 3 4，就違反了「攻 2 前先攻 3」 （2 跑到 3 前面）⇒ 不合法。

字典序最小必須在滿足依賴的前提下逐格取最小，而不是事後排序。這正是 min-heap Kahn 在做的事。

A5 · 正確性（交換論證）

設 v = 目前所有 indeg=0 的點中最小者。對任何合法序 S：v 無前置， 把 v 抽出移到最前面仍合法，而第一格從 S₁ 變成更小的 v ⇒ 字典序更小。 所以最佳序的第一格必是 v。

固定第一格後移除 v、更新入度，剩下的子問題結構完全相同 ⇒ 第二格同理取「現在可選的最小」⇒ 一路歸納到底。因此「每步取堆頂最小」逐格都是最優選擇 ⇒ 整體字典序最小，不只是區域最優。

A · App 1

第一組 n=3，邊 0→1, 1→2, 0→2：indeg = [0,1,2]。

// 第一組
heap={0} → pop 0, 解鎖 1(indeg 1→0)、2(indeg 2→1)  heap={1}
pop 1, 解鎖 2(indeg 1→0)  heap={2}
pop 2  heap={}
輸出 = 0 1 2   （唯一合法序）

第二組 0→1, 1→2, 2→0：三點互相依賴成環，indeg 全為 1，沒有點能進堆 ⇒ order.size()=0 ≠ 3 ⇒ 輸出 QAQ。

A · App 2

輸入規模 m×T 可達 1.5×10⁶ 行。沒關 sync_with_stdio(false) 時 cin 跟 C 的 stdio 同步， 讀入會慢數倍；用 endl 每次強制 flush 緩衝區，輸出也會嚴重變慢 ⇒ 很可能 TLE。

對策：ios::sync_with_stdio(false); cin.tie(nullptr); 開頭加上，輸出一律用 '\n'。

A · App 3

要字典序最大只需把小根堆改成大根堆，只動一行： 

// 字典序最大
priority_queue<int> pq;   // 預設就是 max-heap
// 其餘邏輯完全不變

交換論證對稱地成立：每步取「目前可選的最大」⇒ 整體字典序最大。判循環的方式也一樣（輸出個數 ≠ n ⇒ 有環）。
