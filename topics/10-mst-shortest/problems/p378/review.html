重點回顧// Concept Recap

關鍵點內容
核心觀察 候選邊 c→d 只能讓路長「1⇝c → d⇝N」。
正向 Dijkstra 從 1 求 d1[v] = 起點到 v 的最短距離。
反向 Dijkstra 建反圖、從 N 求 dN[v] = v 到終點的最短距離。
查詢公式 ans = min(d1[N], d1[c] + 1 + dN[d])，O(1)。
獨立性 每個 query 各自假設、互不影響，不能一起塞圖。
數值 距離 ~2×10¹⁴ ⇒ long long；不可達先判 < INF。

觀念題// Concept Questions

Q1 核心建模

為什麼任何「用到候選邊 c→d」的最短路一定長「1⇝c → d⇝N」？怎麼把它拆成兩段預處理？

Q2 反圖

dN[v]（v 到 N 的距離）為什麼要在反圖跑，而不是在正圖從 N 跑？

Q3 獨立性

能不能把全部 Q 條候選邊一起加進圖、只跑一次最短路就回答所有 query？為什麼？

Q4 用幾次

最短路有沒有可能「用候選邊兩次」而更短？用邊權非負說明為什麼只需考慮用一次。

Q5 資料型別

INF 為什麼要設到 4×10¹⁸ 這麼大，而不是 10⁹ 或 LLONG_MAX？

範例 Trace · 跑一次 code 看會輸出什麼// Worked Trace

用樣本 1：N=6, M=5, Q=3，一條鏈 1→2→3→4→5→6，每段長 1。

// INPUT6 5 3
1 2 1
2 3 1
3 4 1
4 5 1
5 6 1
2 5      ← query 1
1 6      ← query 2
3 2      ← query 3

STEP 1 · 兩次 Dijkstra：

v123456
d1[v] (1→v)012345
dN[v] (v→6)543210

base = d1[6] = 5（原最短路）。

// QUERY 1 · (c=2, d=5)
cand = d1[2] + 1 + dN[5] = 1 + 1 + 1 = 3
ans  = min(base=5, 3) = 3
（借 2→5 把中段抄捷徑：1→2 →(新)5 →6）
// QUERY 2 · (c=1, d=6)
cand = d1[1] + 1 + dN[6] = 0 + 1 + 0 = 1
ans  = min(5, 1) = 1
（1→6 直達，整條路只剩長度 1）
// QUERY 3 · (c=3, d=2)
cand = d1[3] + 1 + dN[2] = 2 + 1 + 4 = 7
ans  = min(5, 7) = 5
（3→2 是往回走，沒幫助 ⇒ 維持原最短路 5）
// OUTPUT3
1
5
觀察：query 3 的候選邊 3→2 是往回，繞回去再前進反而更長（7 > 5）， 所以 min 保住原最短路 5。候選邊只有「有幫助才採用」，這個取 min 自動處理。

應用題// Quick Calculations

App 1 手算

承上圖，若候選邊改成 1→5，答案是多少？改成 4→2 呢？

App 2 複雜度

若改成「每個 query 真的把邊加進去重跑 Dijkstra」，總複雜度多少？Q=10⁵ 時大約幾次運算？

App 3 變形

若候選邊長度不是固定 1，而是各自給定 lᵢ，程式要改哪裡？複雜度會變嗎？

// ANSWERS

先自己想過再對答案 — Q 對應 Q、App 對應 App。

A1 · 核心建模

因為候選邊只有一條 c→d。任何用到它的 1→N 路徑，必然在某處走過這條邊， 把路徑切成「到達 c 之前」和「離開 d 之後」兩段。

  • 「1 ⇝ c」這段在原圖，最短就是 d1[c]。
  • 「d ⇝ N」這段在原圖，最短就是 dN[d]。
  • 中間那條候選邊長 1。

三者相加 d1[c] + 1 + dN[d] 就是「一定用這條候選邊」的最短路。預處理 d1[]、dN[] 各一次即可。

A2 · 反圖

軌道是有向的。dN[v] 要的是「沿著邊的方向，從 v 能走到 N 的最短距離」。 在正圖從 N 跑 Dijkstra 求的是「N 出發能到的點」，方向相反、意義不對。

把每條邊 a→b 反向成 b→a 建反圖，再從 N 跑一次 —— 反圖上「N 到 v」的距離，正好等於原圖「v 到 N」的距離。一次跑完拿到所有 dN[]。

A3 · 獨立性

不行。題目是「各別評估每種方案」—— 每個 query 都是「只加這一條候選邊」的假設情境，彼此互不影響。

若把 Q 條邊全加進圖再跑一次，等於同時擁有所有捷徑，會算出比任何單一方案都短的錯誤答案。 正確做法是預處理 d1[]、dN[] 後對每條候選邊分別套公式。

A4 · 用幾次

若一條路徑用候選邊兩次，代表它離開 d 之後又繞回 c、再走一次 c→d —— 中間那段「d ⇝ … ⇝ c」加上兩條候選邊形成一個環。

邊權非負 ⇒ 任何環的長度 ≥ 0 ⇒ 把這個環整段刪掉、改成只用一次，路徑只會更短或一樣。 所以最佳解一定可以「恰用一次」達成，公式只考慮用一次即可。

A5 · INF 大小

合法距離最大約 2×10⁵ 段 × 10⁹ = 2×10¹⁴，所以 INF 必須遠大於這個值， 取 10⁹ 會跟真實距離混淆 ⇒ 太小不行。

又因為公式有 d1[c] + 1 + dN[d]，若直接拿 LLONG_MAX 當 INF， 「INF + 1 + INF」會溢位變負數。取 4×10¹⁸：夠大能當哨兵、相加也不爆 long long（上限 ~9.2×10¹⁸）。再加上「先判 < INF 才相加」更穩。

A · App 1

用 d1=[0,1,2,3,4,5]、dN=[5,4,3,2,1,0]、base=5：

  • 1→5：cand = d1[1]+1+dN[5] = 0+1+1 = 2 ⇒ ans = min(5,2) = 2。
  • 4→2：cand = d1[4]+1+dN[2] = 3+1+4 = 8 ⇒ ans = min(5,8) = 5（往回走沒用）。
A · App 2

每 query 重跑一次 Dijkstra 是 O(M log N)，Q 個就是 O(Q · M log N)。

Q=10⁵、M=5×10⁵、log N≈18 ⇒ 約 10⁵ × 5×10⁵ × 18 ≈ 10¹² 次運算，必定 TLE。 這正是為什麼要改成「兩次預處理 + 每 query O(1)」的 O(M log N + Q)。

A · App 3

把公式中間的常數 1 換成該候選邊的長度 lᵢ 即可： 

// 長度可變
cand = d1[c] + l_i + dN[d]   // 原本的 +1 改成 +l_i

其餘（兩次 Dijkstra 預處理、取 min、判不可達）完全不變。複雜度不變，仍是 O(M log N + Q) —— 預處理跟候選邊長度無關。
