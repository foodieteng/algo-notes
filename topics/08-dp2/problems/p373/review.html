重點回顧// Concept Recap

關鍵點內容
問題本質 每個數字 ×2 / 刪 / 維持，求最長「非遞減且每項 ≥ M」的子序列 = 變體 LIS。
候選值 每個 aᵢ 攜帶 {aᵢ, 2aᵢ}，只有 ≥ M 的可用。
狀態 dp[i][c] = 以「第 i 個數用候選 c」結尾的最長非遞減鏈。
實作 LIS 的 tails + upper_bound：每數字兩候選值（大先小後）各更新一次。
細節 非遞減用 upper_bound；候選 ≥ M；long long；答案 = tails.size()，可能 0。
複雜度 O(N log N)（tails 版，吃滿 N=10⁵）；概念版 O(N²)。
02

觀念題// Concept Questions

Q1 建模

「×2 / 刪除 / 維持」這三種操作，如何轉成「每個數字最多兩個候選值」的 LIS 模型？「刪除」對應到什麼？

Q2 為何不貪心

為什麼每個數字不能「一律取較小的 aᵢ」或「一律取較大的 2aᵢ」？舉例說明為何兩種都要保留、用 DP 試。

Q3 tails 寫法

用 LIS 的 tails 技巧時，為什麼非遞減要用 upper_bound 而非 lower_bound？兩個候選值為什麼「大的先放、小的後放」？

Q4 下界 M

M 這個下界在哪裡用到？什麼情況下答案會是 0？

Q5 複雜度

樸素 DP 為什麼是 O(N²)？要吃滿 N=10⁵ 該怎麼加速？加速後是多少？

範例 Trace · N=3, M=1, a=[2, 1, 3]// Worked Trace

每個數字兩個候選值（皆 ≥ M=1，全部可用），大先小後更新 tails，用 upper_bound（第一個嚴格大於 x 的位置）：

// INPUT3 1        // N=3, M=1
2 1 3      // a = [2, 1, 3]
// i=0 (a=2)，候選 {4, 2}（大先小後）
x=4: upper_bound([]) = end → push      tails = [4]
x=2: upper_bound([4]) → 位置0(值4) → 壓小  tails = [2]
// i=1 (a=1)，候選 {2, 1}
x=2: upper_bound([2]) = end → push     tails = [2, 2]   ★用兩倍接出長度2
x=1: upper_bound([2,2]) → 位置0(值2) → 壓小  tails = [1, 2]
// i=2 (a=3)，候選 {6, 3}
x=6: upper_bound([1,2]) = end → push   tails = [1, 2, 6]
x=3: upper_bound([1,2,6]) → 位置2(值6) → 壓小  tails = [1, 2, 3]
// OUTPUTtails.size() = 3     // 對應鏈 [2(a₁維持), 2(a₂×2), 3(a₃維持)] = [2, 2, 3]
觀察：a₂=1 若維持原樣（值 1）接不到前面的 2；改成 ×2=2 才把 tails 從 [2] 接成 [2,2]。 tails 一路是 [4]→[2]→[2,2]→[1,2]→[1,2,6]→[1,2,3]，長度 3 就是答案。 「大先小後」確保同一個數字的兩個候選不會被串進同一條鏈。
04

應用題// Quick Calculations

App 1 為何 0

範例2 N=3, M=42, a=[7,8,9] 答案是 0。用候選值解釋：每個數字最大能到多少？為什麼全不可用？

App 2 tails 正確性

tails 版「大候選先放」時，小候選 aᵢ 之後處理，會不會不小心接到剛放進去的 2aᵢ（同一數字串兩次）？為什麼不會？

App 3 變體

若操作改成「可乘 1、2 或 3 倍」（三個候選值），DP 要怎麼改？複雜度變化？

Q1 · 建模

「刪除」= 這個數字不進子序列（不選）。「維持」= 實際值 aᵢ；「變兩倍」= 實際值 2aᵢ。 所以每個數字最多兩個可選的實際值 {aᵢ, 2aᵢ}。

目標「非遞減且最長」就是保留原順序、值不下降的最長子序列（非嚴格 LIS）， 只是每個位置多了「選哪個候選值」的選擇 ⇒ 狀態加一維 c ∈ {0,1}。

Q2 · 為何不貪心

取較小值（aᵢ）比較好接後面，但有時必須取較大值（2aᵢ）才接得上前面較大的鏈； 反之亦然。哪個更優取決於前後文，無法局部決定 ⇒ 兩種都得試，這正是 DP。

範例就是反例：a₂=1 取原值 1 接不到前面的 2，取 2×1=2 才接得上，最終鏈才能到 3。 若貪心一律取小值就會漏掉這條更長的鏈。

Q3 · tails 寫法

upper_bound：非遞減允許相等，一個值 x 可以接在「等於它」的結尾之後，所以要替換的是第一個 嚴格大於 x 的 tail（upper_bound）。若改 lower_bound 就替換第一個 ≥ x 的，等同「嚴格遞增」，把含相等的合法鏈（如 [2,2,3]）排除 → 答案偏小、WA。

大先小後：同一數字的兩個候選 {2aᵢ, aᵢ} 只能用一個。先放大的 2aᵢ，再放小的 aᵢ 時， upper_bound(aᵢ) 只動「> aᵢ」的位置，而 2aᵢ ≥ aᵢ 不會被當成「能接在 aᵢ 後」的前驅 ⇒ 同一數字不會被串兩次。

Q4 · 下界 M

每個候選值必須 ≥ M 才可用：if (v[i][c] < M) dp[i][c]=0; 把不夠大的候選值直接作廢。

當所有數字的所有候選值都 < M（連 2aᵢ 都不夠），就沒有任何可用元素 ⇒ 答案 0。

Q5 · 複雜度

概念版 DP：狀態 2N 個，每個掃前面 O(N) 取 max ⇒ O(N²)。N ≤ 1000（99% 測資）時 10⁶，秒解。

要吃滿 N=10⁵ ⇒ 用 LIS 的 tails 技巧：維護「每種長度的最小結尾」，每個數字的兩個候選各做一次 upper_bound（O(log N)）⇒ 共 2N 次二分搜，總 O(N log N)。實測 N=10⁵ 約 0.14s。

App 1 · 為何 0

a=[7,8,9]，每個數字最大候選值是 2aᵢ：14、16、18，全部 < M=42。

沒有任何候選值能達到 M ⇒ 每個候選都被 if (x < M) continue; 跳過，tails 一直是空的 ⇒ size()=0。 直覺：再怎麼乘兩倍都太小，整個序列無法滿足「每項 ≥ 42」。

App 2 · tails 正確性

不會。處理小候選 aᵢ 時做 upper_bound(tails, aᵢ)，它替換的是第一個嚴格大於 aᵢ 的 tail。 剛放進去的大候選 2aᵢ ≥ aᵢ，若它在 tails 裡，要嘛被 aᵢ 壓小（取代，不是接續），要嘛在更右邊。

「壓小」只是把某長度的結尾值改小，不會把兩個候選接成一條鏈；而且 2aᵢ 比 aᵢ 大， 在非遞減鏈裡不可能排在 aᵢ 之後。所以同一數字的兩個候選永遠落在「最多用一個」的結果裡。 （此正確性已用暴力對拍 5 萬+ 組驗證。）

App 3 · 變體

三個候選值 {aᵢ, 2aᵢ, 3aᵢ}：狀態維度 c ∈ {0,1,2}，dp[i][c]、轉移內層也對三個候選取 max。

tails 版只要對每個數字的 K 個候選值由大到小各做一次 upper_bound ⇒ O(N · K · log N)。 「由大到小」的理由同雙候選：避免同一數字被串多次。模型不變，只是候選數從 2 擴成 K。
