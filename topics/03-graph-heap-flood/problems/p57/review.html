重點回顧// Concept Recap

關鍵點內容
結構 完全二元樹存進 1-indexed 陣列；不用指標。
索引 左子 2i、右子 2i+1、父 ⌊i/2⌋。
heap 性質 min-heap：父 ≤ 兩子 ⇒ 最小值永遠在 heap[1]。
insert 放尾端 → sift-up：比父小就上浮，O(log n)。
pop-min 輸出根 → 尾端搬到根 → sift-down：與較小子比、往下沉，O(log n)。
空 heap pop 前先檢查 sz == 0，輸出 empty!。

// ANSWERS

先自己想過再對答案 — Q 對應 Q、App 對應 App。

Q1 · 為何 O(log n)

heap 是完全二元樹，n 個節點高度只有 ⌊log₂ n⌋。 sift-up 每步往父走一層、sift-down 每步往子走一層，最多走樹高那麼多層，每層只做常數比較 / 交換 ⇒ O(log n)。

Q2 · sift-down 細節

因為要維持完全二元樹的形狀：移除的是根，但「空缺」必須由樹最後一個位置來補， 才不會在中間留洞。所以把最後一個元素搬到根（形狀立刻正確），再用 sift-down 修正大小關係。

若直接把某個子提上來，會在那個子的位置留下新空缺，得遞迴處理、且最後一層可能破洞 ⇒ 形狀不再「完全」。

Q3 · 較小子

min-heap 下沉必須跟兩子中較小者交換，換上去的才會 ≤ 另一個子，維持「父 ≤ 兩子」。 

// 換錯（跟左子換）的反例
根 = 9，左子 = 5，右子 = 2
若跟左子 5 換：新根 = 5，但右子 2 < 5 ⇒ 違反 heap 性質！
正確：跟較小的右子 2 換 → 新根 2 ≤ 5 ✓
Q4 · 全序 vs 局部序

heap 不保證陣列整體有序 — 它只保證「每個父 ≤ 它的子」這種局部關係。 例如 [_,1,5,3,8,6] 是合法 min-heap，但陣列不是排序的。

正因為只維持局部序，插入 / 取出只要修一條根到葉的路徑（O(log n)）， 不必像排序陣列那樣維持全序（insert O(n)）。「剛好夠用的弱保證」就是 heap 快的原因。

Q5 · 效能

cin/cout 預設與 C 的 stdio 同步，每次讀寫都有額外成本；10⁶ 次操作（含輸出）會把這成本放大到可能 TLE。 

// 修法
ios::sync_with_stdio(false);
cin.tie(nullptr);
// 輸出用 '\n' 不要 endl（endl 每次 flush）
App 1 · max-heap

手刻：把所有比較的 < 改成 >（sift-up「比父大就上浮」、sift-down「跟較大子換」），根就變成最大值。

STL 更簡單：priority_queue<int> pq;（預設就是 max-heap，top() 是最大）。 反之 min-heap 才需要 greater<int>。

App 2 · 建堆

有。把 n 個數先原封不動放進陣列，再從 i = n/2 往 1 對每個節點做一次 sift-down （build_heap）。

雖然單次 sift-down 是 O(log n)，但靠近底層的節點很多卻只需走很少層，數學上加總是 O(n)， 比逐個 push 的 O(n log n) 快。（本題操作是穿插進來的，用不到一次建堆，但值得知道。）

App 3 · 應用
  • Dijkstra 最短路：用 min-heap 每次取出「目前距離最小」的點。
  • Prim 最小生成樹：用 heap 取出最小權邊。
  • Top-K / 第 K 大、合併 K 條已排序串列、Huffman 編碼、事件模擬的時間佇列…都靠 heap。
