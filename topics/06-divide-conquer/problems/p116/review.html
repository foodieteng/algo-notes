重點回顧// Concept Recap

關鍵點內容
列峰嚴格遞增 row 1..N 的峰欄位嚴格遞增 — 這條結構性條件是分治能成立的唯一理由。
欄位範圍縮減 scan 完 mid 得到 m ⇒ 上半搜尋 [colLo, m]、下半 [m, colHi]。
遞迴順序 ≠ 輸出順序 遞迴是 mid-first，先 row 2 再 row 1 再 row 3 ... — 必須先 answer[] 全填好、最後依序 Report。
GetVal 預算 ≤ (N + M) log₂ N，N = M = 10⁶ 時約 4 × 10⁷ < 10⁸ 上限。
複雜度 T(N, M) = O((N + M) log N)。每層欄位總長 O(M)，共 O(log N) 層。
不需特判 n=1 rowLo > rowHi 自然 return，遞迴收斂乾淨。

// ANSWERS

先自己想過再對答案 — Q 對應 Q、A 對應 A。

A1 · 為何不能 binary search

Binary search 要求被搜尋的序列單調（or 至少 unimodal）。但題目沒保證每列的值有任何排序性質 — 74 59 36 28 51 既不單調也不 unimodal（先降後升）。盲目 binary 不會收斂到峰。

本題的單調性是「跨列」的（列峰欄位嚴格遞增），不是「列內」的。 分治利用的就是這條跨列性質：scan 完 mid 後切上下半，欄位範圍被一刀切兩段，這才是有效的縮減。

對照 P507（最近點對）：那題用 D&C 切的是「點集」按 x 軸切半 — 也是跨元素的切割，不是某個值內部的 binary search。

A2 · 輸出順序

遞迴順序是 mid-first，不是 1..N。4 列的例子 dnc(1, 4, 1, 5)：

  • ① mid = 2 ⇒ scan row 2，先填 answer[2]。
  • ② 上半遞迴 dnc(1, 1, ...)，mid = 1 ⇒ scan row 1，填 answer[1]。
  • ③ 下半遞迴 dnc(3, 4, ...)，mid = 3 ⇒ scan row 3，填 answer[3]。
  • ④ dnc(4, 4, ...)，mid = 4 ⇒ scan row 4，填 answer[4]。

scan 順序：2, 1, 3, 4。輸出順序：1, 2, 3, 4。 所以不能在 scan 完當下 Report，必須先 answer[] 全部填好， solve() 結尾再用 for i in [1, N]: Report(answer[i]) 收尾。

A3 · 欄位端點

嚴格意義上 p_i ≤ m − 1，所以傳 [colLo, m − 1] 完全正確。 但實作上傳 [colLo, m] 有兩個好處：

  • 程式對稱：上半 [colLo, m]、下半 [m, colHi] — 端點都是 m，易讀易維護。
  • 退化邊界乾淨：若 m == colLo（mid 列的峰就在最左），[colLo, m − 1] 會變成空區間 [colLo, colLo − 1]，需要寫保護。[colLo, m] 永遠至少 1 格，不會出錯。

代價：每層上半最多多看 1 個欄位（端點 m 那格），共 O(log N) 次多餘 GetVal — 對總預算 4 × 10⁷ 完全可忽略。

A4 · GetVal 上限

固定某一遞迴層級 k。該層所有子問題的欄位範圍是 [colLo_1, colHi_1], [colLo_2, colHi_2], ..., [colLo_t, colHi_t]。

關鍵觀察：相鄰子問題的欄位範圍只在端點 m 重疊 1 格（因為下半 [m, ...]、上半 [..., m] 共用 m）。所以該層欄位總長 = (colHi_1 − colLo_1 + 1) + ... + (colHi_t − colLo_t + 1) ≤ M + t，而該層子問題數 t ≤ 2^k ≤ N。

該層 GetVal 次數 = 欄位總長 ≤ M + N。共 log₂ N 層 ⇒ 

// TOTAL
GetVal ≤ (N + M) · log₂ N

N = M = 10⁶:  (10⁶ + 10⁶) · 20 = 4 × 10⁷  ✓
N = 10⁴, M = 10⁶: (10⁶ + 10⁴) · 14 ≈ 1.4 × 10⁷  ✓
A5 · 弱化條件

如果條件改成 p_i ≥ p_{i−1}（允許相等），目前的演算法直接成立，不需任何修改：

  • 上半遞迴範圍 [colLo, m] — 端點 m 已經包含進去，所以 p_i = m 的情況也找得到。
  • 下半遞迴範圍 [m, colHi] — 同理。

其實本題的程式就是用「≤ m / ≥ m」寫的，比題目嚴格保證的「< m / > m」還寬。 嚴格遞增只是讓正確性的推導更乾淨；演算法本身對「不嚴格遞增」一樣對。

反過來：若條件是「列峰任意分布」（無單調性），分治就垮了 — 沒有任何根據可以縮減上下半的欄位範圍。那種情況只能退化回 O(N·M) 暴力。

A · App 1

套公式 (N + M) log₂ N： 

// N = 10^4, M = 10^6
log₂(10^4) ≈ 13.3 (取 14)
(10^4 + 10^6) · 14 ≈ 10^6 · 14 = 1.4 × 10^7

比 N = M = 10⁶ 的 4 × 10⁷ 還小。原因：log₂ N 從 20 降到 14，而 M 沒變、N 又遠小於 M。 所以 N 小不等於慢 — 真正的瓶頸是 M。

A · App 2

N = 1 時 dnc(1, 1, 1, M)：

  • mid = 1，scan(1, 1, M) 掃整列，填 answer[1]。
  • dnc(1, 0, ...) 與 dnc(2, 1, ...) 都因 rowLo > rowHi 而直接 return。
  • 最後 Report(answer[1])。

正常運作，不需特判。遞迴的 base case 本來就是 rowLo > rowHi 自然 return，不依賴 N 的大小。

題目雖然保證 N ≥ 2，但程式對 N = 1 也是對的 — 加分項。

A · App 3

程式不會 crash — 它每列都會掃出該列在「縮減後欄位範圍內」的最大值，並寫進 answer[]。 但結果不一定對：

  • 若實際 row 1 的峰在 col 3、row 2 的峰在 col 1（違反嚴格遞增）：
  • scan row 2 (cols 1..5) ⇒ 找到 col 1 是 row 2 真正的峰，m = 1。
  • 上半遞迴 [colLo, 1] = [1, 1] ⇒ row 1 只能掃 col 1 — 看不到 col 3 的真峰。

所以輸出會是「行為未定義的錯誤答案」（WA）。 演算法的正確性完全依賴「列峰嚴格遞增」這條題目保證，judge 若違反它就出題出錯，不是 user 的責任。
