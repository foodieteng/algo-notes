01

重點回顧// Concept Recap

關鍵點內容
分治結構 切半 → 各自遞迴排序 → merge 時順便算跨界逆序對的兩數和。
跨界對的數法 merge 取右半 a[j] 時，左半剩 count = mid − i + 1 個全部 > a[j]，全部成對。
貢獻公式 contrib = remaining + a[j] × count — 左半剩餘和 + a[j] 乘上左半剩餘個數。
remaining 維護 merge 開始算一次左半總和；左半游標前進就 remaining −= a[i]，O(1) 取用。
複雜度 T(N) = 2·T(N/2) + O(N) = Θ(N log N)（Master Theorem case 2）。
Mod 規則 累加進 inv_sum、remaining 都 mod；比較永遠用原始值。

應用題// Quick Calculations

A1 最壞情況

N = 10⁶ 全降序時，逆序對數量有多少？這也是為什麼不能 O(N²)。

A2 特例

對已遞增的序列 [1, 2, 3, 4, 5] 跑 code，inv_sum 是多少？code 還需要跑完整 merge sort 嗎？

A3 相關題

若把題目改為「對每組逆序對輸出 a[i] − a[j] 之和」，code 要改哪幾行？

// ANSWERS

先自己想過再對答案 — Q 對應 Q、A 對應 A。

A1 · 跨界數法

因為左半在 merge 之前已經排序過，所以 a[i] ≤ a[i+1] ≤ … ≤ a[mid]。 既然 a[i] > a[j]（觸發取右），那 a[i..mid] 全部 > a[j]。

再者左半的「原始下標」全部 < 右半（合併時保留左在前）， 所以每組 (a[p], a[j])（p ∈ [i, mid]）都滿足 「下標小者 > 下標大者」⇒ 都是逆序對。

共 mid − i + 1 個元素，每個都跟 a[j] 配成一對 ⇒ count 個逆序對一次算完。

A2 · 不重不漏

任一逆序對 (p, q) 滿足 p < q 且 a[p] > a[q]。 把它按下標歸類到三種情境：

  • 若 p、q 都在左半 ⇒ 在左半遞迴時算到。
  • 若 p、q 都在右半 ⇒ 在右半遞迴時算到。
  • 若 p 在左、q 在右 ⇒ 在「這層」的 merge_range 中算到。

三種情境互不相交且每對恰好在一處被算進 ⇒ 三者相加即為總和，無重複也無遺漏。

A3 · 複雜度

Master Theorem 形式 T(N) = a · T(N/b) + f(N)，本題 a = 2, b = 2, f(N) = O(N)。

logb a = log₂ 2 = 1，且 f(N) = Θ(N¹)， 屬 case 2（f(N) = Θ(Nlogb a)） ⇒ T(N) = Θ(Nlogb a · log N) = Θ(N log N)。

直覺：recursion tree 共 log₂ N 層，每層所有 merge 加起來掃過 N 個元素 ⇒ 總工時 Θ(N log N)。

A4 · Mod 陷阱

mod 把不同的大數對映到相同的餘數，破壞順序。 例如 MOD = 7、a = 9、b = 2： 

// COMPARE
原始比較：9 > 2           → 是逆序對 ●
mod 後  ：9 % 7 = 2, 2 % 7 = 2
         2 > 2 為 false  → 誤判為非逆序對 ×

排序與比較必須用原始值。mod 只作用在「累加進 inv_sum」與「remaining 的維護」這兩處。

A5 · 型別

remaining 雖然 mod 後 < 10⁷ 可用 int，但中間運算 (remaining + (a[j] % MOD) × (count % MOD)) 的乘法 10⁷ × 10⁶ ≈ 10¹³，遠超 32-bit int（≤ 2.1 × 10⁹）。 

// INT OVERFLOW EXAMPLE
a[j] % MOD = 9_999_999 (≈ 10⁷)
count      = 1_000_000 (= 10⁶)

int:       9_999_999 × 1_000_000  ⇒ 溢位 (int 最大 ≈ 2.1 × 10⁹)
long long: 9_999_999 × 1_000_000 = 9.999... × 10¹²  ● 安全

a[i] 本身 ≤ 10⁹ 也逼近 int 上限。一律宣告 long long 最保險。

A · App 1

全降序時，每一對 (i, j)（i < j）都是逆序對。

數量 = C(N, 2) = N(N − 1)/2。 N = 10⁶ ⇒ 10⁶ × (10⁶ − 1)/2 ≈ 5 × 10¹¹ 對。

這正是不能 O(N²) 暴力的原因 — 5 × 10¹¹ 次運算需要好幾分鐘到好幾小時，而 merge sort 的 O(N log N) ≈ 2 × 10⁷ 次運算秒過。

A · App 2

[1, 2, 3, 4, 5] 已嚴格遞增 ⇒ 沒有任何逆序對 ⇒ inv_sum = 0。

但 code 仍會跑完整的 O(N log N) merge sort — 每個 merge step 都進「取左、無貢獻」分支， inv_sum 從頭到尾維持 0。我們沒有用「是否已排序」做提前終止，因為這需要額外檢查。

若想加速，可以在 merge_range 開頭加 if (a[mid] <= a[mid+1]) { 直接 copy 兩半到 tmp 並 return; } — 但這個常數優化在隨機資料上沒什麼幫助。

A · App 3

把 a[i] + a[j] 改為 a[i] − a[j]：左半剩 count 個都 > a[j]，每個都貢獻 a[i] − a[j]。

合起來的貢獻 = (a[i] + a[i+1] + … + a[mid]) − a[j] × count = remaining − a[j] × count。

code 改動：

  • contrib = remaining − a[j] × count（原本是 +）。
  • 因為 remaining − a[j]×count 可能是負數（mod 後也可能負），要寫成 (remaining − (a[j] % MOD) × (count % MOD) % MOD + MOD) % MOD 才安全。
  • 其他結構（雙指針、remaining 維護、copy back）完全不變。

核心觀念依然是「取右半 a[j] 時，左半剩 count 個全部跟它配對」，只是貢獻公式換了。
