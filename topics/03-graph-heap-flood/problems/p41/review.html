重點回顧// Concept Recap

關鍵點內容
問題本質 數「水的 4-連通塊」個數 = connected components。
主迴圈 row-major 掃全圖，遇到沒走過的水才 count++ 並 flood。
Flood Fill BFS / 迭代 DFS 把整灘水標記 visited；一灘只被「開新池」一次。
入列即標記 push 進 queue 的當下就 visited = true，避免同格重複入列。
避免遞迴 1000×1000 全水 ⇒ 遞迴深 10⁶ 爆 stack；改用全域 queue 陣列。
複雜度 O(H × W) 每筆 — 每格造訪 O(1) 次。

範例 Trace · 跑一次 code 看會輸出什麼// Worked Trace

用題目範例第 4 筆地圖（4×4，正解 = 3）。座標以 (row, col) 0-indexed：

r \ c0123
0.###
1#..#
2#..#
3###.

輸入：

// INPUT (這一筆)4 4
.###
#..#
#..#
###.

主迴圈 row-major 掃描，pools 初始 0：

// SCAN 到 (0,0) = '.' 且未走過 → 開新池
pools: 0 → 1
flood(0,0):
  push (0,0), visited[0][0]=true
  pop (0,0): 上(出界) 下(1,0)='#' 左(出界) 右(0,1)='#' → 無新格
  queue 空 → pool 1 = { (0,0) }
// SCAN (0,1)(0,2)(0,3) = '#' → 跳過
// SCAN (1,0) = '#' → 跳過
// SCAN 到 (1,1) = '.' 且未走過 → 開新池
pools: 1 → 2
flood(1,1):
  push (1,1)
  pop (1,1): 上(0,1)# 下(2,1). → push  左(1,0)# 右(1,2). → push
  pop (2,1): 上(1,1)v 下(3,1)# 左(2,0)# 右(2,2). → push
  pop (1,2): 上(0,2)# 下(2,2)v 左(1,1)v 右(1,3)#
  pop (2,2): 上(1,2)v 下(3,2)# 左(2,1)v 右(2,3)#
  queue 空 → pool 2 = { (1,1),(2,1),(1,2),(2,2) }  ← 中間 2×2 整塊
// SCAN (1,2)(2,1)(2,2) 已走過 → 跳過；其餘為 '#'
// SCAN (1,3)(2,0)(2,3)(3,0)(3,1)(3,2) = '#' → 跳過
// SCAN 到 (3,3) = '.' 且未走過 → 開新池
pools: 2 → 3
flood(3,3):
  push (3,3)
  pop (3,3): 上(2,3)# 下(出界) 左(3,2)# 右(出界) → 無新格
  queue 空 → pool 3 = { (3,3) }
// OUTPUT3
觀察：中間 4 格水 (1,1)(1,2)(2,1)(2,2) 雖然各自被掃到， 但只有最先碰到的 (1,1) 觸發 pools++，其餘三格在掃到時都已是 visited ⇒ 一灘只數一次。這正是計數正確的核心。

應用題// Quick Calculations

App 1 最大水池面積

若題目改問「最大那灘水池有幾格」，code 要加什麼？

App 2 八方向

若相鄰改成含對角線（8 連通），要改哪裡？答案會變大還變小？

App 3 記憶體

H = W = 1000 時，qx[]、qy[] 各要開多大？用 int 大約吃多少記憶體？

// ANSWERS

先自己想過再對答案 — Q 對應 Q、App 對應 App。

Q1 · 計數正確性

因為 count++ 和「flood 淹滿整灘」是綁在一起的：每次 +1，緊接著的 BFS 就把同一灘所有格標成 visited。所以同一灘裡，只有主迴圈最先碰到的那一格是「未走過的水」、才會 +1； 其餘格之後掃到時都是 visited ⇒ 不重複數。又因 row-major 一定走遍全圖，每灘至少被碰一次 ⇒ 不漏數。兩者合起來 count 恰好 = 連通塊數。

Q2 · DFS vs BFS

遞迴 DFS 邏輯正確，但遞迴深度 = 最壞連通塊大小。 1000 × 1000 全是水時，一灘就有 10⁶ 格 ⇒ 遞迴深 10⁶， 遠超預設呼叫堆疊（通常幾 MB / 約 10⁴∼10⁵ 層）⇒ Stack Overflow / RE。

BFS（或手動 stack 的迭代 DFS）把待辦格子放在全域陣列（heap）， 深度再大也只是陣列變長，不碰呼叫堆疊 ⇒ 安全。

Q3 · 入列時機

必須在入列（push）的當下就 visited = true。 

// 若改成「出列時才標記」
某格 (a,b) 是兩個已在 queue 中的格子的共同鄰居
→ 處理第一個鄰居時 (a,b) 未 visited → push
→ 處理第二個鄰居時 (a,b) 仍未 visited（還沒輪到出列）→ 又 push
⇒ 同格被 push 多次 → queue 膨脹、可能超出陣列上界 → RE

入列即標記，保證每格恰好 push 一次。

Q4 · 複雜度

每格水恰好 push 一次、pop 一次（入列即標記保證）。pop 後看固定 4 個方向 ⇒ 每格 O(1)。 所有 flood 加總處理的格數 = 水格總數 ≤ H × W ⇒ 整段 BFS 是 O(H × W)，不是 O((HW)²)。

主迴圈掃描也是 O(H × W)，兩者相加仍 O(H × W)。

Q5 · 多筆測資

每筆之間要重置 visited（這份 code 在讀地圖時逐格清成 false）。 grid 因為整張重讀會被覆蓋、不必特別清；qx/qy 靠 head/tail 在每次 flood 開頭歸零，自然不殘留。

若漏清 visited：上一筆標過的格子在這一筆仍被當「走過」⇒ 該數的水池被跳過 ⇒ 答案偏小。

App 1 · 最大水池面積

讓 flood 回傳這次淹了幾格（BFS 結束時 tail 即為格數）， 主迴圈用 best = max(best, flood(i,j)) 取最大。輸出 best 即可，同樣 O(H×W)。

App 2 · 八方向

把方向陣列從 4 個擴成 8 個（加四個對角）： 

// 8 連通
int dx[8] = {-1,-1,-1, 0, 0, 1, 1, 1};
int dy[8] = {-1, 0, 1,-1, 1,-1, 0, 1};
// 迴圈上界 d < 8

連通性變強（更多格被視為相鄰）⇒ 水池更容易連在一起 ⇒ 水池數只會變少或不變，不會變多。

App 3 · 記憶體

最壞整圖全水，一次 flood 要裝 H × W = 10⁶ 個座標 ⇒ qx[]、qy[] 各要能放 10⁶ 個 int。

  • 每個 int 4 bytes ⇒ 單個陣列 10⁶ × 4 B ≈ 4 MB，兩個約 8 MB。
  • 再加 grid（char，~1 MB）、visited（bool，~1 MB），總計約 10 MB — 在常見 256 MB 限制內安全。

所以 code 用 qx[MAXN*MAXN] 開到 ≈ 10⁶ 是必要且足夠的。
