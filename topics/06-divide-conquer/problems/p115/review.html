[ 5.1 ] Concept Recap

重要概念回顧

關鍵點內容
分治結構 把陣列切兩半 → 各自遞迴排序 → merge 時順便算跨界逆序對。
跨界逆序對 merge 時取 a[j]（右半）出來，左半剩 a[i..mid] 個都 > a[j]，全部成對。
貢獻公式 contrib = remaining_left_sum + a[j] × count，其中 count = mid − i + 1。
remaining 維護 merge 開始時算一次左半總和，左半游標每前進一步就減掉 a[i]，O(1) 取用。
時間複雜度 T(N) = 2·T(N/2) + O(N) ⇒ Θ(N log N)（Master Theorem case 2）。
Mod 必要 累加進 inv_sum 與 remaining 都 mod；比較永遠用原始值。
[ 5.2 ] Concept Questions

觀念題

Q1 分治

在 merge 中取右半的 a[j] 出來時，為什麼左半剩餘的 a[i..mid] 全部跟 a[j] 形成逆序對？

Q2 構造

為什麼總逆序對和 = 左半內部逆序對和 + 右半內部逆序對和 + 跨界逆序對和？這三者為何不會重複計算？

Q3 複雜度

用 Master Theorem 推導 T(N) = 2·T(N/2) + O(N) = Θ(N log N)。

Q4 Mod

為什麼比較大小不能用 a[i] % MOD 後的值？舉一個會出錯的例子。

Q5 實作

remaining 與 contrib 為什麼要宣告成 long long？用 int 會在什麼地方爆掉？

[ 5.4 ] Quick Calculations

應用題

A1 最壞情況

N = 10⁶ 全降序時，逆序對的數量是多少？

A2 配分

為什麼 70% 配分上限訂在 N ≤ 45,678 而不是更小？

A3 特例

對已排序的序列 [1, 2, 3, 4, 5] 跑 code，inv_sum 是多少？

// ANSWERS

先自己想過再對答案。下面按題號編排。

A1 · 分治

因為左半在 merge 之前已經排序過，所以 a[i] ≤ a[i+1] ≤ … ≤ a[mid]。 又 a[i] > a[j]（觸發取右），所以 a[i..mid] 全部 > a[j]。 再者左半的「原始下標」全部 < 右半的「原始下標」（合併時保留 left 在前）， 所以 (a[p], a[j]) 對於 p ∈ [i, mid] 都滿足 「下標小者 > 下標大者」⇒ 逆序對。

A2 · 構造

任一逆序對 (p, q) 滿足 p < q 且 a[p] > a[q]。 把它依下標歸類到「左半」「右半」「跨界」三種情境：

  • 若 p、q 都在左半 ⇒ 在左半內部遞迴時被算到。
  • 若 p、q 都在右半 ⇒ 在右半內部遞迴時被算到。
  • 若 p 在左、q 在右 ⇒ 在「合併這層」的 merge 中被算到。

三種互不相交，且每對恰好在一處被算進。因此三者相加即為總數， 無重複也無遺漏。

A3 · 複雜度

Master Theorem 形式 T(N) = a·T(N/b) + f(N)，本題 a = 2, b = 2, f(N) = O(N)。

計算 log_b a = log₂ 2 = 1，而 f(N) = Θ(N^1)。 這屬 case 2（f(N) = Θ(N^{log_b a})）⇒ T(N) = Θ(N^{log_b a} · log N) = Θ(N log N)。

直覺：recursion tree 共 log₂ N 層，每層所有 merge 加起來掃過總共 N 個元素 ⇒ 總工時 Θ(N log N)。

A4 · Mod

mod 會把不同的大數對映到相同的餘數，破壞順序。 例如 MOD = 7、a = 9、b = 2： 

原始比較：9 > 2     → 是逆序對 ✓
mod 後比較：9 % 7 = 2、2 % 7 = 2
          2 > 2 為 false → 誤判為非逆序對 ✗

排序與比較必須用原始值，mod 只對 inv_sum 累加與 remaining 維護生效。

A5 · 實作

remaining 雖然 mod 後 < 10⁷ 可用 int，但中間運算 (remaining + (a[j] % MOD) × (count % MOD)) 的乘法 10⁷ × 10⁶ ≈ 10¹³，遠超 32-bit int（≤ 2.1 × 10⁹）。 int 在乘法當下就溢位。改用 long long 才安全。

a[i] 本身 ≤ 10⁹，也已逼近 int 上限。一律 long long 最保險。

A · Ex 1

[3, 1] 只有一次 merge：左 [3]、右 [1]。 

remaining = 3
i = 0, j = 1
a[0] = 3, a[1] = 1
3 > 1 ⇒ count = 1
contrib = 3 + 1 × 1 = 4

inv_sum = 4

結果 inv_sum = 4（對應到唯一一個逆序對 (3,1)，兩數和 4）。

A · Ex 2

對 [4, 2, 3, 1] 跑 merge sort：

merge_sort(a, 0, 3)
  merge_sort(a, 0, 1)         遞迴左半 [4, 2]
    merge_sort(a, 0, 0)       base
    merge_sort(a, 1, 1)       base
    merge(a, 0, 0, 1):
      remaining = 4
      4 > 2 ⇒ contrib = 4 + 2×1 = 6
    ▸ inv_sum = 6
    a = [2, 4, 3, 1]

  merge_sort(a, 2, 3)         遞迴右半 [3, 1]
    merge_sort(a, 2, 2)       base
    merge_sort(a, 3, 3)       base
    merge(a, 2, 2, 3):
      remaining = 3
      3 > 1 ⇒ contrib = 3 + 1×1 = 4
    ▸ inv_sum = 10
    a = [2, 4, 1, 3]

  merge(a, 0, 1, 3):           合併左半 [2,4] 與右半 [1,3]
    remaining = 2 + 4 = 6
    a[0]=2, a[2]=1: 2 > 1 ⇒ count = 2, contrib = 6 + 1×2 = 8
  ▸ inv_sum = 18
    a[0]=2, a[3]=3: 2 ≤ 3 ⇒ 取左，remaining 變 4
    a[1]=4, a[3]=3: 4 > 3 ⇒ count = 1, contrib = 4 + 3×1 = 7
  ▸ inv_sum = 25
    a = [1, 2, 3, 4]

最終 inv_sum = 25。 驗算：所有逆序對 (4,2)=6、(4,3)=7、(4,1)=5、(2,1)=3、(3,1)=4 ⇒ 6+7+5+3+4 = 25 ✓

A · Ex 3

MOD = 7、[10, 5]：

merge(a, 0, 0, 1):
  比較用原始值：10 > 5 ⇒ 是逆序對
  remaining = 10 % 7 = 3
  contrib = (3 + (5 % 7) × (1 % 7)) % 7
          = (3 + 5 × 1) % 7
          = 8 % 7
          = 1

inv_sum = 1

注意比較永遠用 10 vs 5（原始），不會用 mod 後的 3 vs 5。

A · App 1

全降序時，每一對 (i, j) with i < j 都是逆序對。

數量 = C(N, 2) = N(N − 1)/2。 N = 10⁶ ⇒ 10⁶ × (10⁶ − 1)/2 ≈ 5 × 10¹¹ 對。

這正是為何不能 O(N²) 暴力 — 5 × 10¹¹ 次運算需要好幾小時。

A · App 2

45,678² ≈ 2.09 × 10⁹，剛好是常數內運算「2-3 秒」能跑完的量級（典型 OJ 時限）。

這是故意給 70% 配分一個 O(N²) 也能勉強過的空間 — 沒寫出 merge sort 解法的人， 還是能用暴力撈到 30% + 部分 70%。100% 必須 O(N log N)。

A · App 3

[1, 2, 3, 4, 5] 已嚴格遞增，沒有任何逆序對。

跑 merge sort 時每個 merge step 都進「取左、無貢獻」分支，inv_sum 維持 0。 程式仍會跑 O(N log N) 的 merge sort，只是不累加任何貢獻。
