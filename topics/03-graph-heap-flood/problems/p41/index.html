庭院裡的水池

// Flood Fill · 連通塊計數經典

PAGE 1 / 3  ·  概念
01

題目簡介// Brief

QUESTION

給一張 H × W 的地圖（H, W ≤ 1000），. 是水、# 是障礙。 一格水與它上下左右相鄰的水算同一灘水池。問地圖上共有幾灘不同的水池？ 多筆測資 T ≤ 10。

換句話說：把每格水當圖的點、相鄰的水之間連邊，問有幾個連通塊（connected components）。 這就是 Flood Fill 的標準應用。

02

△ REJECT · STACK OVERFLOW

C BFS Flood Fill O(HW)
掃過每格，遇到沒走過的水就 +1 並用 queue 淹滿整灘、全部標記為走過。每格只進 queue 一次。
● ADOPT · STABLE

DFS 本身沒錯，遞迴版才危險。要嘛改成手動 stack 的迭代 DFS，要嘛用 BFS（queue） — 兩者都把「待處理的格子」放在 heap 記憶體而非呼叫堆疊，徹底避開爆 stack。本卡採 BFS。

03

核心策略 · 三步驟// Algorithm Outline

維護一個 visited[H][W]，主迴圈 row-major 掃過每一格：

  1. 掃描 · 遇到 # 或已走過的水 → 跳過。
  2. 開新池 · 遇到沒走過的水 → 答案 +1，把它丟進 queue。
  3. BFS 淹滿 · 不斷從 queue 取格，把四方向沒走過的水標記 visited 並推進 queue，直到 queue 空 → 這一整灘處理完。

關鍵不變量：每格水恰好被標記一次，所以總工作量 = 格子數 = O(H × W)。

05

為什麼剛好數對// Correctness

// INVARIANT · 每灘恰好被「開新池」一次
  1. 主迴圈只在遇到沒走過的水時才 count++。
  2. 緊接著的 BFS 會把同一灘所有水標記成 visited。
  3. 所以同一灘裡，只有最先被主迴圈碰到的那一格會觸發 count++；其餘格之後都因「已走過」而跳過 ⇒ 不重複數。
  4. 每格水遲早會被主迴圈掃到（row-major 一定走遍全圖）⇒ 每灘至少數到一次 ⇒ 不漏數。
  5. ⇒ count 恰好等於連通塊個數。

一句話：count++ 與「淹滿一整灘」綁在一起，一灘只觸發一次。

06

複雜度// Complexity

主迴圈走過 H × W 格各一次；BFS 部分，每格水最多進 queue 一次、出 queue 一次， 每次看 4 個方向 ⇒ 整個 BFS 的總工作量也是 O(H × W)。

T(H, W) = O(H × W)  每筆測資
