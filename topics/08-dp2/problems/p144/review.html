重點回顧// Concept Recap

關鍵點內容
問題本質 0/1 背包 + 「最多 K 件」⇒ 多開一維記已選個數。
狀態 dp[j][l] = 選 ≤ j 個、飽足 ≤ l 的最大滿足感。
轉移 dp[j][l] = max(dp[j][l], dp[j−1][l−Aᵢ]+Bᵢ)。
來源 j−1 吃一個 ⇒ 用掉 1 個名額 + Aᵢ 飽足 ⇒ 從 dp[j−1][l−Aᵢ] 轉移。
兩維倒序 j、l 都由大到小，保證每物只拿一次。
複雜度 O(N·K·M) 時間、O(K·M) 空間；答案 dp[K][M]。
02

觀念題// Concept Questions

Q1 為何多一維

原題（P143）的一維 dp[l] 為什麼沒辦法處理「最多 K 個」？要記什麼才補得起來？

Q2 轉移推導

轉移式裡「吃」這項為什麼是 dp[j−1][l−Aᵢ]+Bᵢ？j−1 和 l−Aᵢ 各對應什麼資源被消耗？

Q3 兩維倒序

為什麼 j 和 l 兩維都要倒序？只有一維倒序、另一維正序會發生什麼？

Q4 答案位置

為什麼答案直接取 dp[K][M]，而不用對 j、l 取 max 掃一遍？

Q5 退化

當 K = N 時這題等於什麼？同一份二維程式還會正確嗎？為什麼？

範例 Trace · M=6, K=3, 物品 (2,4)(3,5)(4,6)// Worked Trace

dp 是 (K+1)×(M+1) 表（列 j = 已選個數、行 l = 飽足）。逐物品 j、l 兩維倒序：

// INPUT3 6 3
2 4         // 物品0：A=2 B=4
3 5         // 物品1：A=3 B=5
4 6         // 物品2：A=4 B=6
// 處理物品0 (A=2,B=4)  每列 l: 6→2
        l=0  1  2  3  4  5  6
j=0  :  0  0  0  0  0  0  0
j=1  :  0  0  4  4  4  4  4
j=2  :  0  0  4  4  4  4  4
j=3  :  0  0  4  4  4  4  4
// 處理物品1 (A=3,B=5)
        l=0  1  2  3  4  5  6
j=0  :  0  0  0  0  0  0  0
j=1  :  0  0  4  5  5  5  5
j=2  :  0  0  4  5  5  9  9    ← j=2,l=5: dp[1][2]+5 = 4+5 = 9（吃物品0+1）
j=3  :  0  0  4  5  5  9  9
// 處理物品2 (A=4,B=6)
        l=0  1  2  3  4  5  6
j=0  :  0  0  0  0  0  0  0
j=1  :  0  0  4  5  6  6  6
j=2  :  0  0  4  5  6  9 10    ← j=2,l=6: dp[1][2]+6 = 4+6 = 10（吃物品0+2）
j=3  :  0  0  4  5  6  9 10
// OUTPUTdp[K=3][M=6] = 10
觀察：答案 10 來自 dp[2][6] = 吃物品0(A=2,B=4) + 物品2(A=4,B=6)：飽足 2+4=6 ≤ 6、用 2 個 ≤ K=3、滿足 4+6=10。 雖然 K 容許 3 個，但第三個塞不進剩餘容量，所以最佳只用 2 個。
04

應用題// Quick Calculations

App 1 K 變小

若同物品但 K=1（只能吃一個），答案是多少？從 trace 的哪一列讀得出來？

App 2 壓維

這題的二維 dp 能不能壓成一維（像 P143）？提示：要保留哪一維、犧牲哪一維？

App 3 恰好 K 個

若改成「恰好吃 K 個」（不是最多），dp 初始化與答案讀取要怎麼改？

Q1 · 為何多一維

一維 dp[l] 只記「容量 l 下的最大價值」，不知道用了幾個物品，所以無法限制「最多 K 個」—— 它會把能塞進容量的全部拿走，可能超過 K 個。

補法：多開一維 j 記「已選個數」，狀態變 dp[j][l]，就能把「件數」也當成一種有限資源來管理。

Q2 · 轉移推導

「吃第 i 個」會同時消耗兩種資源：用掉 1 個名額（j → j−1）與 Aᵢ 飽足（l → l−Aᵢ）。

所以「吃之後的 dp[j][l]」要從「吃之前、少一個名額又少 Aᵢ 飽足」的狀態 dp[j−1][l−Aᵢ] 轉移，再加上這顆的 Bᵢ。 和「不吃」（保留舊 dp[j][l]）取大。

Q3 · 兩維倒序

轉移用到的 dp[j−1][l−Aᵢ] 必須是「還沒考慮第 i 個物品」的值（上一輪的表），第 i 個才不會被重複拿。

j 倒序讓 dp[j−1] 本輪還沒被改；l 倒序讓 dp[·][l−Aᵢ] 本輪還沒被改。 若任一維正序，對應的來源格已被本輪更新過、可能已含第 i 個 ⇒ 同物品重複計入，答案偏大、錯。所以兩維都倒序。

Q4 · 答案位置

因為 dp 定義是「選 ≤ j 個、飽足 ≤ l」（不是恰好），這個「≤」讓 dp 對 j、l 都是單調不減的—— 放寬上限不會讓最佳值變小。

所以最寬鬆的格子 dp[K][M] 已經涵蓋所有「≤ K 個、≤ M 飽足」的選法，直接取它即可，不必再掃 max。

Q5 · 退化

K = N 表示「最多吃 N 個」= 沒有件數限制（本來就只有 N 個可吃）⇒ 退化成原題 P143 的純 0/1 背包。

同一份二維程式仍正確：j 維可以一路加到 N 都不受限，dp[N][M] 等於一維背包的答案。只是多花了 O(K) 倍時間（K=N），對 N≤100 無妨。

App 1 · K=1

K=1 ⇒ 只能吃一個，答案讀 dp[1][6]。從 trace 最後一張表的 j=1 那列： dp[1][6] = 6（單吃物品2，A=4≤6、B=6 最大）。 

// j=1 列（最後狀態）
l :  0  1  2  3  4  5  6
   :  0  0  4  5  6  6  6      → dp[1][6] = 6
App 2 · 壓維

可以壓掉「物品 i 這一層」（本來就只用滾動陣列、沒展開成 dp[i][..]），但 j 和 l 兩維都得保留—— 它們是真正的狀態（已選個數、已用飽足），少了任一維就無法表達限制。

也就是說空間已經是 O(K·M)（最省），不能再壓成一維；倒序的寫法本身就是「物品維被滾動掉」的結果。

App 3 · 恰好 K 個

改「恰好」要區分「不可達」狀態：初始化 dp[0][0]=0、其餘全設 −∞（表示「還沒湊到那個個數」是不合法的）。 轉移不變。

答案讀 dp[K][l] 對 l = 0..M 取 max（恰好 K 個、飽足任意 ≤ M 的最大值）； 若 dp[K][·] 全是 −∞ 代表湊不出 K 個，輸出無解。本題是「最多」所以用 0 初始化、直接 dp[K][M]，較單純。
