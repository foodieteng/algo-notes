重點回顧// Concept Recap

關鍵點內容
構造而非搜尋 不枚舉、不 DP，直接造一條保證合法的排列。
奇偶切原則 所有奇數放前段、所有偶數放後段；段內遞迴。
跨段零 3-AP 3-AP 兩端必同奇偶 ⇒ 已被物理分離 ⇒ 跨段不可能。
段內歸納 段內變成規模一半的同一個問題，由歸納假設仍 3-AP free。
映射公式 左段 2x − 1 映成奇數、右段 2x 映成偶數。
複雜度 T(N) = T(⌈N/2⌉) + T(⌊N/2⌋) + O(N) = O(N log N)。

範例 Trace · build(out, 5)// Worked Trace

用 N = 5 跑一次 build()，看遞迴怎麼一層一層回上來：

// INPUT5

遞迴展開的樹狀結構 — 葉節點都是 build(_, 1)：

// CALL TREE
             build(_, 5)
           /            \
       build(_, 3)    build(_, 2)
       /       \      /       \
   build(_,2) build(_,1)  build(_,1) build(_,1)
    /    \
 build(_,1) build(_,1)

⌈5/2⌉ = 3 (左)、⌊5/2⌋ = 2 (右)
⌈3/2⌉ = 2 (左)、⌊3/2⌋ = 1 (右)
⌈2/2⌉ = 1 (左)、⌊2/2⌋ = 1 (右)

由葉節點往上回傳（每一層都用映射公式 2x − 1 接 2x）：

// CALL 1 · build(_, 1)
out[0] = 1
return → [1]
// CALL 2 · build(_, 2)
leftLen = 1, rightLen = 1
build(buf, 1)              → buf = [1]
out[0] = 2·1 − 1 = 1       ← 左映成奇
build(buf, 1)              → buf = [1]
out[1] = 2·1     = 2       ← 右映成偶
return → [1, 2]
// CALL 3 · build(_, 3)
leftLen = 2, rightLen = 1
build(buf, 2)              → buf = [1, 2]
out[0] = 2·1 − 1 = 1
out[1] = 2·2 − 1 = 3
build(buf, 1)              → buf = [1]
out[2] = 2·1     = 2
return → [1, 3, 2]
// CALL 4 · build(_, 5) — 主呼叫
leftLen = 3, rightLen = 2
build(buf, 3)              → buf = [1, 3, 2]
out[0] = 2·1 − 1 = 1
out[1] = 2·3 − 1 = 5
out[2] = 2·2 − 1 = 3
build(buf, 2)              → buf = [1, 2]
out[3] = 2·1     = 2
out[4] = 2·2     = 4
return → [1, 5, 3, 2, 4]
// OUTPUT1 5 3 2 4

驗證一下：N = 5 共 C(5, 3) = 10 個三元組 (i, j, k)，逐一比 2·a[j] 與 a[i] + a[k]：

(i, j, k) (a[i], a[j], a[k]) 2·a[j] vs a[i] + a[k] OK?
(1, 2, 3)(1, 5, 3)10 ≠ 4 ●
(1, 2, 4)(1, 5, 2)10 ≠ 3 ●
(1, 2, 5)(1, 5, 4)10 ≠ 5 ●
(1, 3, 4)(1, 3, 2)6 ≠ 3 ●
(1, 3, 5)(1, 3, 4)6 ≠ 5 ●
(1, 4, 5)(1, 2, 4)4 ≠ 5 ●
(2, 3, 4)(5, 3, 2)6 ≠ 7 ●
(2, 3, 5)(5, 3, 4)6 ≠ 9 ●
(2, 4, 5)(5, 2, 4)4 ≠ 9 ●
(3, 4, 5)(3, 2, 4)4 ≠ 7 ●
觀察：10 / 10 全通過 — 證明跟程式輸出一致。 注意每組 (a[i], a[k]) 都是同奇偶或異奇偶，但 a[j] 偏偏沒站在「平均」的位置上 — 這就是奇偶切的力量。

應用題// Quick Calculations

App 1 遞迴深度

N = 10⁶ 時 build 的最大遞迴深度是多少？需要擔心 stack overflow 嗎？

App 2 記憶體配置

為什麼 code 用兩個全域陣列 seq[]、buf[]，而不是在 build() 內部開區域陣列或 vector？

App 3 變形題

若改題目「避免任意 4 項等差」(4-AP free)，奇偶切還適用嗎？要怎麼修改思路？

// ANSWERS

先自己想過再對答案 — Q 對應 Q、App 對應 App。

A1 · 3-AP 性質

3-AP 條件 2·a[j] = a[i] + a[k] ⇒ 右式 a[i] + a[k] 為偶數 ⇒ a[i] 與 a[k] 同奇偶（奇+奇=偶、偶+偶=偶；奇+偶=奇排除）。

奇偶切把 1..N 按奇偶物理分離到前段與後段。要找跨段 3-AP，必有 i 在前段、k 在後段 ⇒ a[i] 是奇數、a[k] 是偶數 ⇒ 違反「同奇偶」必要條件 ⇒ 跨段不存在 3-AP。 只剩同段內部的 3-AP — 而段內就是規模一半的同一個問題，由歸納假設仍 3-AP free。

A2 · 切半比例

1..N 中奇數有 ⌈N/2⌉ 個、偶數有 ⌊N/2⌋ 個。 所以左段（奇數段）長度必須是 ⌈N/2⌉。

  • N 偶（如 N = 6）：⌈6/2⌉ = ⌊6/2⌋ = 3，左右各半，寫反沒差。
  • N 奇（如 N = 5）：奇數 {1, 3, 5} 有 3 個、偶數 {2, 4} 有 2 個。 若把左段寫成 ⌊N/2⌋ = 2，左段只裝得下兩個奇數，第三個奇數沒地方放、第二個偶數位置也錯亂 — 結果不是 1..N 的排列。

正確寫法：leftLen = (len + 1) / 2; 即 ⌈len/2⌉。

A3 · 構造法 vs 搜尋
方法複雜度能跑到
枚舉 + 驗證O(N! · N³)N ≤ 10 左右
狀態 DP (子集)O(2ᴺ · N)N ≤ 24
奇偶切構造O(N log N)N = 10⁶ 級

前兩者本質是搜尋空間—「N! 種排列裡找一個合法的」、或「2ᴺ 個子集記憶哪些前綴可行」； 構造法則是直接給出公式，不必探索 — 這就是構造類題的標誌： 有合法結構可被精確描述時，造出來比找出來快得多。

A4 · 合法答案非唯一

仍合法。把偶數放前、奇數放後（或左右互換）對應原構造的鏡射版本， 同樣可由奇偶切的證明保證 3-AP free — 證明本身對前後段對稱。

實際上合法答案非常多 — 例如 N = 5 時 [1, 5, 3, 2, 4]、[5, 1, 3, 4, 2]、 [2, 4, 1, 5, 3] 等等都合法。題目只要求一個合法答案。

A5 · 複雜度
  • Time · T(N) = T(⌈N/2⌉) + T(⌊N/2⌋) + O(N) = O(N log N)。Master Theorem case 2。
  • Space · O(N) — 兩支固定大小的全域陣列；遞迴 stack O(log N)。
  • Recursion depth · O(log N)。

合併工作每層 O(N)（而非 O(1)）的原因： 每一層都要把 buf 內的子問題答案 for-loop 映射 (2x − 1 / 2x) 並複製到 out 的對應段。 左段 ⌈len/2⌉ 個元素、右段 ⌊len/2⌋ 個元素，合起來 = len 個元素的線性 work。

App 1 · 遞迴深度

log₂(10⁶) ≈ 19.9，所以最深約 20 層。 現代 OS 預設 stack 約 1–8 MB，每層 stack frame 約幾十 bytes，20 層完全沒壓力。

但若題目 N 來到 10⁹+ 級（雖然這題不會），就要改成 iterative 版本 — 用 size 1 → 2 → 4 → 8 … bottom-up 倍增、或對 buf 與 out 做 in-place 變換。

App 2 · 記憶體配置

幾個原因：

  • 避免動態配置 overhead — N = 10⁶ 時若每層 new/delete 或建 vector，常數變大且可能爆 fragmentation。
  • 共用緩衝 — buf[] 在「解左→映左」與「解右→映右」之間可以被覆蓋，因為左段答案已寫到 out。整個遞迴只需一支 buf。
  • 避免 stack 上開大陣列 — 局部變數 int local[len] 在 len 大時直接爆 stack。全域陣列在 BSS 段、無此問題。

這是構造類遞迴題的標準寫法 — 全域陣列 + 指標傳遞 + 共用 scratch buffer。

App 3 · 4-AP free 變形

奇偶切對 4-AP 不直接適用。 4-AP a[i], a[j], a[k], a[l] 滿足等差 d，端點 a[i], a[l] 不必同奇偶 — 例如 (1, 3, 5, 7) 全奇但仍是 4-AP。奇偶切只能阻擋「兩端同奇偶」這條 3-AP 必要條件，對 4-AP 沒效。

正確思路要改：4-AP free 問題與 Behrend 構造（在 ℤ/Nℤ 中找大的 3-AP free 集合，再 lift 到整數）有關， 建構更困難 — 目前已知最大 4-AP free 集合大小是 N^(1 − ε(N))，沒有像奇偶切這麼乾淨的遞迴公式。

延伸閱讀：Behrend 1946、Szemerédi 1975（任意長度等差自由集合的上界）— 此屬組合數學 / 加性組合範疇，超出競賽常見範圍。
